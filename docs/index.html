Substitutes Agent Demo

A small implementation of retail substitute mapping on open apparel data. Built as an agentic workflow, with an experiment on where LLM judgment breaks.

Repo on GitHub · Vastra, my related project

The raw data

The Kaggle apparel dataset ships 44,446 rows. Before any code runs, the data has the usual mess.

  • Category leakage. 2,404 Personal Care rows, 25 Sporting Goods, 1 Home. All arrive inside an apparel dataset.
  • No brand column. Brand is buried inside productDisplayName. "Peter England Men Party Blue Jeans" needs a heuristic to become brand = Peter England.
  • Long tail everywhere. 316 rows with usage NA. 21 rows with empty season. Colors like Skin, Charcoal, Off White with no clean parent family.
  • Category imbalance. Apparel 21,400. Accessories 11,289. Footwear 9,222. Everything else tiny.

One raw row looks like this:

15970, Men, Apparel, Topwear, Shirts, Navy Blue, Fall,
2011, Casual, Turtle Check Men Navy Blue Shirt

Step 1. Filter and normalize

Deterministic. No LLM. Pure Python.

  • Filter to Apparel + Footwear. 44,446 becomes 30,622.
  • Drop rows with missing brand-derivable name, article type, or color.
  • Derive brand heuristically from productDisplayName leading tokens.
  • Dedupe by product id.
  • Normalize brand casing.

The brand heuristic is honest about its failure modes. "United Colors of Benetton" truncates to "united colors of". "Red Tape" collapses to "red". "Gini and Jony" survives by accident. This flows directly into the pair classifier. Documented, not hidden.

What got filtered out

13,824 rows left the dataset before Step 2. The biggest cut is category leakage: 11,289 Accessories, 2,404 Personal Care, 105 Free Items, 25 Sporting Goods, 1 Home. A deodorant, a football, a cushion cover, a free tie. None of them belong in an apparel substitute map. The rest are rows with missing brand-derivable names, article types, or colours, plus 22 malformed rows the loader reconstructed from ragged CSV lines.

Step 2. Build the product ontology

Rules first. LLM only where the rules leave a row ambiguous.

For each SKU the pipeline extracts:

  • article_type, aligned to a Vastra-adapted taxonomy
  • master_category
  • base_colour, mapped to a family (navy and teal both cluster to blue)
  • usage (Casual, Formal, Sports, Ethnic)
  • gender
  • season
  • pattern (regex: Solid, Striped, Printed, Checked, Floral)
  • material (regex: Cotton, Denim, Linen, Silk, Polyester, Leather)

The Vastra tie-in

Vastra is my AI wardrobe app. Its categorizer uses a small stable vocabulary and its prompt passes the tags already in the wardrobe so the model reuses spellings instead of inventing new ones. Same pattern here: the LLM path passes patterns and materials already observed in this catalog run, so the model aligns to a stable vocabulary.

The LLM only runs if a row has both pattern and material null despite a long product name. Results are cached to disk so re-runs are free.

Committed output: output/step2_ontology.json

Step 3. Classify pairs

Blocking, scoring, and a narrow LLM tie-break band.

Blocking rule

Two SKUs are only considered a candidate pair if they share master_category, article_type, and gender. This turns 124,750 naive pairs into 7,429 candidates on the 500-row sample. A 94 percent reduction before the scoring layer runs.

Three-way classification

  • VARIANT. Same brand, same article, same colour, same usage. Different size or season is fine.
  • SUBSTITUTE. Different brand, same article, colour family overlap of at least 0.5, same usage, same gender.
  • UNRELATED. Everything else. Not emitted.

Scoring formula

total_score =
    0.35 * article_type_match
  + 0.20 * colour_similarity
  + 0.15 * usage_match
  + 0.10 * pattern_match
  + 0.15 * material_similarity
  + 0.05 * season_overlap

LLM tie-break

Only fires when total_score sits in [0.45, 0.55] and a provider key is set. On this run the band was empty and the LLM tie-break fired zero times. Rules alone resolved every pair. The LLM shows up elsewhere, in Step 5.

Reading the brand table

Fabindia at 77.78 percent gap makes sense. Indian ethnic wear is genuinely hard to substitute out of. Nike at 38.24 and Adidas at 28.21 are branded sportswear with partial private-label overlap. Puma, Wrangler, and Locomotive at zero percent gap have tight segments where every SKU has at least one substitute.

Reading the near-miss table

The blocking reason is not what I expected before I ran the numbers. I assumed the colour-family gate would dominate, and it does for four of the ten (pink vs blue, yellow vs blue, red vs black, brown vs purple). But six of ten are blocked by a usage mismatch: a Formal shirt sitting next to a Casual shirt in the same blocking group, or an Ethnic top next to a Casual one. They block together on article type and gender, but the SUBSTITUTE rule correctly refuses to bridge Formal and Casual. That is the rule doing its job, not a gap to fix.

Step 5. Consistency experiment

Zenline's Q2 2026 update argues that raw autonomous agents give inconsistent answers to the same question. This step tests that claim on this project's own ambiguous pairs.

Interpretation

Not sure what to make of the within-model consistency. Gemini agrees with itself on every pair, Claude on 29 of 30, but the interesting number is cross-model agreement at 80%. For a retailer that reframes the failure mode: it's not "ask the same LLM twice, get two answers" (less of a problem at the small tier than the rhetoric suggests), it's "swap vendors and watch the substitute map shift" on some fraction of borderline SKUs.

How big that fraction is depends on how you deploy. One query per pair with no retry and 20% of borderline pairs have at least one disagreeing verdict across the 9 runs. Majority vote across three runs per model, which is what any rational production system would do, and only 1 pair in 30 (3.3%) actually flips. Every other cross-model disagreement is within-model noise on a single model, most often GPT-4o-mini. That one pair is the worked example below.

20% is the ceiling under naive deployment, 3.3% the floor under sensible retry. Either way the rules pipeline handles the deterministic majority before an LLM sees it, so drift only ever applies to the borderline residual. A vendor change becomes a versioned diff on a small auditable subset, not a silent shift across the whole map.

Where this fits

Four papers that shaped what I built or chose not to.

Ma et al. (Aalborg, Amsterdam, TU Berlin), CE-RAG4EM. arXiv:2602.05708. Feb 2026.

Blocking plus LLMs is the economically viable pattern at scale. My Step 3 does the same, minus the external knowledge-graph lookup. They beat supervised PLMs on 6 of 9 standard entity-matching benchmarks.

Zhang, Khan, Walter (Amazon), CatalogRAG. LLM4ECommerce Workshop at KDD'25. Aug 2025. Deployed across Amazon US, DE, FR stores.

BM25 retrieval and brand-aware reranking feed LLM prompts with in-catalog few-shot examples. My Step 2 LLM path is a scrappier version of the same intuition. They chose BM25 over vector embeddings for operational simplicity, the same reason I use rules instead of embeddings.

Tang et al. (USTC, Alibaba), Large Reasoning Embedding Models. arXiv:2510.14321. Oct 2025. Deployed at Taobao since Aug 2025.

LLM reasoning baked into the embedding layer to bridge the semantic gap on hard queries. I did not take that direction: my retrieval is deterministic blocking on structured attributes, not semantic search.

Kothari et al. (Flipkart), semantic retrieval for product search. arXiv:2606.01504. May 2026.

Production Siamese dual-encoder with four classes: Perfect Match, Substitute, Complementary, Irrelevant. Maps onto my VARIANT, SUBSTITUTE, UNRELATED, minus the Complementary class. What you build when you have labeled substitute data. I don't.

Why I built this

Because it is also a good test for Phase 2 of Vastra and where I'd take my own project next. Building this validated something we'd hit in Vastra's internal prototype testing: model changes meant inconsistent tagging on the same garment. Vastra's own architecture already has a pipelined flow where agents are called in for verification and tagging but do not make autonomous decisions, precisely because of these inconsistencies. Zenline's public argument for agentic workflows lands on the same conclusion from the retailer side of the aisle, which is why the demo was worth building.

Honest limitations

  • No ground truth in this demo, though public benchmarks exist. There are no labeled "these are actual substitutes" annotations on the paramaggarwal Kaggle dataset. Two public alternatives could be dropped in for evaluation: the Amazon Shopping Queries Dataset (Reddy et al., 2022) with ESCI labels (Exact, Substitute, Complement, Irrelevant), and the Magellan and WDC entity-matching benchmarks used in Ma et al. (2026). I did not use them for this demo because the paramaggarwal dataset is messy and proabbly closer to the raw-retailer-catalog + it is garment dataset, making it easy for me to "port" over things from Vastra. Adopting one of those benchmarks would be the obvious first extension to close the evaluation loop.
  • English-only, on a single dataset. The paramaggarwal dataset is primarily English with some transliterated terms. Zhang et al. (2025) demonstrate that in-catalog few-shot approaches yield the largest gains on non-English stores, where LLM base performance is weaker. Extending this pipeline to a multilingual retailer would likely make the Step 2 LLM path more valuable, not less.
  • The paramaggarwal dataset is sourced from myntra.com and is volunteer/catalog-contributed; it is sparse in places and India-skewed.
  • The dataset has no dedicated brand column. I derive brand heuristically from the leading tokens of productDisplayName (consume until a boundary token — gender / usage / colour / article noun — capped at 3). It is imperfect by design; mis-derived brands flow directly into the VARIANT/SUBSTITUTE rules.
  • The rules-based ontology is deliberately simple: a small regex vocabulary for pattern and material, and a hand-built colour-family map. ~72% of sample rows have neither pattern nor material extracted and are flagged low-confidence — exactly the gap the optional LLM path is meant to lift.
